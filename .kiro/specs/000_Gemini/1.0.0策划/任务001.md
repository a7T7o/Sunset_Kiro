## 二、核心叙事锚点（已确定方向，你需要深度扩展）

主角是一名【失忆的穿越者】。他穿越并坠落在一个被世人遗弃的、极其落后破败的边境村落。

关键设定：
- 主角失忆，不知道自己是谁，以"旅人"身份被村民收留和接纳
- 主角并非从零学习，而是"回忆"——随着任务触发、与村民交互、在地牢中接触特定遗物，主角会发生"记忆闪回"，从而觉醒（解锁）高阶的工作台图纸与物品配方
- 终极目标：利用觉醒的知识，从最原始的手工开始，一步步建立自动化生产，最终振兴这个破败的村落，并在过程中揭开自己身世的秘密

你需要基于这个方向，深度扩展：
- 主角的身世背景（穿越前是谁？为什么穿越？为什么失忆？）
- 村落的具体设定（有哪些建筑？哪些NPC？为什么被遗弃？）
- 主线剧情的完整脉络（从被收留到振兴村落的全过程）
- 支线任务和事件设计（每个NPC的任务线、季节事件、随机事件）
- "记忆闪回"的具体触发机制和叙事表现

叙事风格要求：
- 不要传统的"继承爷爷的农场"老套路
- 可以带有一些新奇、搞怪、幽默的元素，但不要强行荒诞
- 故事要服务于玩法，每一个叙事设定都应该对应一个游戏机制
- 要有情感锚点——玩家应该对村民产生感情，对村落的复兴有成就感

## 二点五、设计哲学与核心铁律（你的所有设计必须遵守）

以下四条铁律是本项目的设计灵魂。你在后续每一个篇章的设计中，都必须反复检查是否违反了这些铁律。违反任何一条，整个设计作废重来。

### 铁律一：星露谷式三维压迫感——约束驱动决策

> 开发者原话："星露谷的上瘾就是每一天的重复但事件不重复，金钱有限精力有限时间有限地图过大只能走路这种时间消耗的影响"

游戏的成瘾性不来自"内容多"，而来自"约束紧"。玩家每天醒来都有强烈的目标感，但一天的时间永远不够用，迫使玩家不断做优先级决策。

三维约束体系：
- 【时间】：跑图慢、晚上2点强制昏倒、季节限定作物/事件制造紧迫感
- 【精力】：干活消耗精力（`ToolData.energyCost` 字段已存在），必须吃料理恢复，精力不够就干不了活
- 【空间/资源】：背包格子有限、金钱有限、材料获取有门槛

设计检查标准：如果玩家在任何一天能"轻松做完所有想做的事"，说明约束设计失败。

### 铁律二：从"折磨"到"解放"的自动化三阶段

自动化是本游戏最核心的成长体验。前期的痛苦是为了让后期的解放感更加强烈。

- **Phase 1 · 手搓期（极度痛苦）**：所有二次加工都需要玩家亲自站在工作台前操作，耗时长、交互繁琐。采集来的木料要在锯木台一根根锯，矿石要在熔炉一块块烧。玩家会强烈感受到"时间不够用"的痛点
- **Phase 2 · 半自动期（初尝甜头）**：通过地牢遗物触发"记忆闪回"，解锁工作台升级图纸。升级后的工作台可以批量处理、缩短耗时。玩家开始感受到效率提升的快感
- **Phase 3 · 全自动期（彻底解放）**：进一步解锁自动进料/出料机制，工作台可以在玩家离开后自动运转。玩家的精力被彻底解放，可以去探索更深的地牢、推进更多的村落复兴任务

设计检查标准：每个工作台（`WorkstationData` 的6种类型）都必须有清晰的三阶段演进路线。

### 铁律三：钥匙背包博弈——本项目的核心亮点

这是面试时最值得讲述的设计亮点，必须在策划案中被重点展现。

核心机制：
- 钥匙不可堆叠（`maxStackSize=1`），每把钥匙占一个背包格子
- 进入地牢深层或开宝箱需要钥匙，带的钥匙越多，能装战利品的格子越少
- 开锁有概率系统（`KeyLockData.unlockChance`）：成功保留钥匙，失败消耗钥匙
- 死亡全掉落：在地牢中血量归零或耗到凌晨2点，物品全部掉落
- 逃脱绳也占格子：保命道具同样挤占背包空间

玩家每次进地牢前的核心决策："我是带3把钥匙去深层赌大件，还是带满食物稳扎稳打？要不要留一格给逃脱绳？"

已有代码支撑：`KeyLockData.cs`（钥匙/锁数据）、`StorageData.cs`（箱子材质/血量/上锁）、`ItemData.maxStackSize`（堆叠控制）、`ItemQuality`（品质影响概率）

设计检查标准：地牢奖励表和钥匙获取途径必须让玩家始终面临"风险vs回报"的纠结。

### 铁律四：叙事服务于玩法——温情成长基调

故事不是装饰品，每一个叙事设定都必须对应一个游戏机制。

基调要求：
- 整体情感基调是温情、成长、陪伴——玩家应该对村民产生感情，对村落的复兴有成就感
- 可以有幽默、新奇、搞怪的元素，但不要黑暗恐怖、不要强行荒诞
- "失忆穿越者"的设定服务于配方解锁机制（回忆觉醒 = 解锁隐藏配方）
- "破败村落"的设定服务于长期目标（村落复兴 = 可量化的里程碑系统）
- NPC不是工具人，每个NPC都应该有让玩家记住的性格特点和故事线

设计检查标准：如果删掉某个叙事设定后玩法不受影响，说明这个设定是多余的，需要重新绑定机制。

## 三、已完成的代码架构（真实数据，逐模块精确描述）

这是一个 2D 俯视角像素风沙盒模拟经营 + 地牢探索游戏。以下是已经落地的代码系统，按模块逐一说明。

### 3.1 核心架构层

**事件总线 EventBus**
- 文件：`Core/Events/EventBus.cs`、`GameEvents.cs`、`IGameEvent.cs`
- 已定义的事件类型：
  - 时间系统：`MinuteChangedEvent`（每10分钟触发）、`HourChangedEvent`、`DayChangedEvent`、`SeasonChangedEvent`、`YearChangedEvent`、`SleepEvent`
  - 天气系统：`WeatherChangedEvent`、`PlantsWitherEvent`（植物枯萎）、`PlantsRecoverEvent`、`WinterSnowEvent`、`WinterMeltEvent`
  - 植被系统：`VegetationSeasonChangedEvent`
  - 导航系统：`NavGridRefreshRequestEvent`（支持指定位置和半径的局部刷新）
  - 背包系统：`InventoryChangedEvent`、`HotbarSelectionChangedEvent`、`ToolEquippedEvent`
  - 玩家系统：`PlayerActionStartEvent`、`PlayerActionCompleteEvent`
  - UI系统：`PanelOpenedEvent`、`PanelClosedEvent`
  - 放置系统：`PlacementEvents.cs`（独立文件）
  - 工具系统：`ToolEvents.cs`（独立文件）

**数据驱动 ScriptableObject 体系**
- 基类：`ItemData`（所有物品的父类）
  - 字段：itemID（0000-9999分段）、itemName、description、category、icon、bagSprite、worldPrefab、buyPrice、sellPrice、maxStackSize（1=不可堆叠）、isPlaceable、placementType、equipmentType、consumableType
  - 品质系统：`ItemQuality` 枚举（Normal/Rare/Epic/Legendary），品质影响售价倍率（1.0/1.25/2.0/3.25）
- 物品ID分段规范：
  - 0XXX = 工具和武器（00XX农业工具、01XX采集工具、02XX武器）
  - 1XXX = 种植类（含种子、树苗、工作台1300-1399、存储容器1400-1499、钥匙1420-1499、锁1410-1419）
  - 2XXX = 动物产品
  - 3XXX = 矿物和材料（30XX矿石、31XX锭、32XX自然材料、33XX怪物掉落）
  - 4XXX = 消耗品
  - 5XXX = 食品
  - 6XXX = 家具
  - 7XXX = 特殊物品

- 子类 `ToolData`（工具数据）：
  - 继承 ItemData，ID范围 0000-0199
  - 字段：toolType（Hoe/WateringCan/Sickle/FishingRod/Pickaxe/Axe）、energyCost、effectRadius、efficiencyMultiplier、hasDurability、maxDurability、materialTier（Wood/Stone/Iron/Brass/Steel/Gold 六级材料等级）、canDealDamage、damageAmount
  - 动画系统：animatorController（直接拖拽赋值）、animActionType（Slice=6挥砍/Pierce=7刺出/Crush=8挖掘/Fish=9钓鱼/Watering=10浇水）、animationFrameCount
  - 动画命名规范：`{ActionType}_{Direction}_Clip_{itemID}_{quality}`
  - 工具不可堆叠（maxStackSize=1）

- 子类 `WeaponData`（武器数据）：
  - 继承 ItemData，ID范围 0200-0299
  - 字段：weaponType（Sword/Bow/Staff/Spear/Hammer）、attackPower、attackSpeed、criticalChance、criticalDamageMultiplier、attackRange、knockbackForce、materialTier、energyCostPerAttack
  - 有 `CalculateDamage()` 方法，内置暴击计算
  - 武器不可堆叠

- 子类 `SeedData`（种子数据）：
  - 继承 ItemData，ID范围 1000-1099
  - 字段：season（Spring/Summer/Fall/Winter/AllSeason）、isReHarvestable、reHarvestDays、maxHarvestCount
  - 种子袋系统：seedsPerBag、shelfLifeClosed、shelfLifeOpened、iconOpened
  - 关联：cropPrefab（作物预制体，必须包含 CropController 组件）
  - 种植需求：needsTrellis（是否需要支架）
  - 经验：plantingExp、harvestingExp
  - 种子标记为 isPlaceable=true，走放置系统

- 子类 `CropData`（作物数据）：
  - 继承 FoodData（不是直接继承 ItemData），ID范围 1100-1149
  - 作物可食用，继承了 FoodData 的 energyRestore、healthRestore、buffType 等字段
  - 数据流是单向的：SeedData → cropPrefab → CropController → dropItemData

- 子类 `WitheredCropData`（枯萎作物数据）：
  - 继承 FoodData，ID范围 1150-1199
  - 枯萎作物可食用但有负面效果
  - 通过 CropController.witheredDropItemData 配置

- 子类 `SaplingData`（树苗数据）：
  - 继承 PlaceableItemData（抽象基类），ID范围 1200-1299
  - 字段：treePrefab（关联的树木预制体，必须包含 TreeController）、plantingExp
  - 冬季无法种植
  - PlaceableItemData 基类提供：placementOffset、canRotate、rotationStep，以及抽象方法 GetPlacementType()、GetPlacementPrefab()

- 子类 `WorkstationData`（工作台数据）：
  - 继承 PlaceableItemData，ID范围 1300-1399
  - 字段：workstationType（Heating取暖/Smelting冶炼/Pharmacy制药/Sawmill锯木/Cooking烹饪/Crafting制作）、recipeListRef、craftTimeMultiplier
  - 燃料系统：requiresFuel、fuelSlotCount、acceptedFuelTypes

- 子类 `StorageData`（存储容器数据）：
  - 继承 PlaceableItemData，ID范围 1400-1499（其中钥匙1420-1499、锁1410-1419 由 KeyLockData 管理）
  - 字段：storageCapacity、storageRows、storageCols、allowedCategories、isLockable、defaultLocked
  - 箱子属性：chestMaterial（Wood/Iron/Special）、maxHealth、baseUnlockChance
  - 预制体：storagePrefab（世界物体）、boxUiPrefab（UI预制体，如 Box_12/Box_24/Box_36/Box_48）

- 子类 `KeyLockData`（钥匙/锁数据）：
  - 继承 ItemData，钥匙ID 1420-1499，锁ID 1410-1419
  - 字段：keyLockType（Key/Lock）、material（ChestMaterial）、unlockChance
  - 开锁概率系统：最终概率 = 钥匙开锁概率 + 箱子被开锁概率；成功保留钥匙，失败消耗钥匙
  - 特殊钥匙/锁：isSpecial、targetChestId

- 子类 `InteractiveDisplayData`（交互展示数据）：
  - 继承 PlaceableItemData，ID范围 1500-1599
  - 字段：displayTitle、displayContent、displayImage、displayDuration、closeOnInteract
  - 用途：告示牌、信息板、纪念碑等

- 子类 `SimpleEventData`（简单事件数据）：
  - 继承 PlaceableItemData，ID范围 1600-1699
  - 字段：eventType（PlaySound/PlayAnimation/Teleport/Unlock/GiveItem/TriggerQuest/ShowMessage）、eventParameter、isOneTime、cooldownTime
  - 用途：传送点、开关、触发器等

- 子类 `FoodData`（食物数据）：
  - 继承 ItemData，ID范围 5000-5999
  - 字段：energyRestore、healthRestore、consumeTime
  - Buff系统：buffType（None/Speed/Defense/Attack/Luck/Fishing/Mining/Farming）、buffValue、buffDuration
  - 配方关联：recipeID

- 子类 `PotionData`（药水数据）：
  - 继承 ItemData，ID范围 4000-4099
  - 字段：healthRestore、energyRestore、useTime
  - Buff系统：同 FoodData
  - 视觉效果：useEffectPrefab、useSound

- 子类 `MaterialData`（材料数据）：
  - 继承 ItemData，ID范围 3000-3999
  - 字段：materialSubType（Ore矿石/Ingot锭/Natural自然材料/Monster怪物掉落）、sourceDescription
  - 熔炼系统：canBeSmelt、smeltResultID（产物ID在3100-3199锭类范围）、smeltTime
  - 用途标记：craftingUse

- 子类 `EquipmentData`（装备数据）：
  - 继承 ItemData
  - 字段：defense、attributes（StatModifier列表）、equipmentModel
  - StatModifier：statType（Strength/Agility/Intelligence/Vitality/Luck/Defense/Attack/MaxHealth/MaxMana/CritChance/CritDamage）× modifierType（Flat/Percent）
  - 装备类型：Helmet/Armor/Pants/Shoes/Ring/Accessory
  - 装备不可堆叠

**配方系统 RecipeData**
- 独立 ScriptableObject，不继承 ItemData
- 字段：recipeID、recipeName、description、resultItemID、resultAmount
- 材料列表：`List<RecipeIngredient>`，每项含 itemID + amount
- 制作条件：requiredStation（None手工/CookingPot烹饪锅/Furnace熔炉/MagicTower魔法塔/AnvilForge铁砧/Workbench工作台/AlchemyTable制药台/Grill烧烤架）、requiredLevel、craftingTime
- 解锁系统：unlockedByDefault、requiredSkillType、requiredSkillLevel、isHiddenRecipe（非等级解锁，通过特殊方式解锁）、unlockCondition
- 奖励：craftingExp

**掉落表 DropTable**
- 独立 ScriptableObject
- 每个 DropConfig：item（ItemData引用）、minAmount、maxAmount、dropChance（0-1概率）、quality
- 生成参数：spreadRadius、guaranteeOneDrop（保证至少掉落一个）


### 3.2 背包与物品系统

**InventoryService**（背包服务）
- 单例，管理玩家背包
- 支持存档（Save/Load）
- 背包格子使用 `ItemStack` 结构：itemId + quality + amount
- ItemStack 支持堆叠判断：相同 itemId + 相同 quality 才能堆叠

**HotbarSelectionService**（快捷栏选择服务）
- 管理玩家当前选中的快捷栏格子
- 发布 `HotbarSelectionChangedEvent` 和 `ToolEquippedEvent`

**ChestInventory / ChestInventoryV2**（箱子背包）
- 独立于玩家背包的存储容器
- 支持不同容量（12/24/36/48格）
- 支持存档

**InventorySortService**（背包排序）
- 自动合并可堆叠物品
- 按物品类型优先级排序

**IItemContainer**（物品容器接口）
- 统一接口：GetSlot、SetSlot、ClearSlot、SwapOrMerge
- 玩家背包和箱子都实现此接口

### 3.3 制作系统

**CraftingService**（制作服务）
- 单例，管理所有配方的解锁和制作
- 监听技能升级事件，自动解锁对应等级的配方
- 制作失败原因枚举 `FailReason`

### 3.4 农田系统

**FarmTileManager**（农田格子管理器）
- 管理多层级农田格子（支持室内/室外不同层级）
- 核心操作：CreateTile（开垦）、RemoveTile（移除）、SetWatered（浇水）、CanTillAt（检查是否可开垦）
- 每日重置浇水状态（ResetDailyWaterState）
- 支持存档（Save/Load）
- 使用 Tilemap 系统渲染农田视觉

**CropController**（作物控制器）
- 挂载在作物预制体上，管理单株作物的完整生命周期
- 状态机：Seed（种子）→ Growing（生长中）→ Mature（成熟）→ Harvested（已收获）
- 枯萎状态：WitheredImmature（未成熟枯萎）、WitheredMature（成熟后枯萎）
- 生长系统：每日检查，根据阶段配置推进生长
- 收获系统：成熟收获（掉落 dropItemData）、枯萎收获（掉落 witheredDropItemData）
- 可重复收获：isReHarvestable 的作物收获后回退到指定阶段重新生长
- 季节检查：非适宜季节的作物会枯萎
- 支持存档

**CropManager**（作物管理器）
- 管理场景中所有作物实例

**FarmVisualManager**（农田视觉管理器）

**TimeManager**（时间管理器）

**SeasonManager**（季节管理器）

**WeatherSystem**（天气系统）

#### 4.5.1 树木素材 — 由 `TreeController` 驱动的完整生命周期系统

**素材结构**（位于预制体配置中，通过 `TreeSpriteConfig` 引用）：

- 树桩：`Stump.png`（砍伐后的树桩通用素材）

**代码驱动方式**（`TreeController` — 2200+行，`TreeSpriteData.cs` 定义数据结构）：
- 6个成长阶段（`STAGE_COUNT = 6`），每阶段独立配置：成长天数、血量、碰撞体、遮挡、掉落表
  - 阶段0（树苗）：用锄头挖出，无碰撞无遮挡，`daysToNextStage=1`
  - 阶段1-2（小树/中树）：用斧头砍伐，`health=4/9`
  - 阶段3-5（大树/成熟/完全成熟）：有树桩系统，`health=17/28/40`，`stumpHealth=9/12/16`
- 5套季节 Sprite 系统（`SeasonSpriteSet`）：Spring / Summer / EarlyFall / LateFall / Winter
  - 季节渐变机制：基于 `SeasonManager.GetTransitionProgress()` 和 `persistentId.GetHashCode()` 生成固定随机种子
  - 渐变不可逆：一旦 `treeSeedValue < progress`，该树就切换到下一季节外观
  - 显示时间线：春1-14纯春 → 春15-28春夏渐变 → 夏1-14纯夏 → 夏15-28夏秋渐变 → 秋1-14早秋晚秋渐变 → 秋15-28纯晚秋 → 冬全程冬季
- 天气响应系统：
