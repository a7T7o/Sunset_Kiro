### 声明 3：种子幽灵占位是 Instantiate 竞态条件 🔴

锐评声称：
> "在极短的帧间隔内，Unity 的 Instantiate (实例化预制体) 和底层数据校验产生了竞态条件（Race Condition）。数据写进去了，但预制体因为某种物理或层级原因实例化失败或者被瞬间销毁了"

**代码事实核查**：


**ExecutePlantSeed 执行流程分析**（第1597-1710行）：

```
入队路径：TryEnqueueSeed → EnqueueAction → _queuedPositions.Add(key) → ProcessNextAction
执行路径：ExecuteFarmAction PlantSeed 分支 → ExecutePlantSeed → RemoveExecutingPreview → _queuedPositions.Remove → ProcessNextAction（立即取下一个）
```

**关键发现 1：种子是同步串行执行的，不存在帧级竞态**

ExecuteFarmAction 的 PlantSeed 分支是**完全同步**的：
```csharp
case FarmActionType.PlantSeed:
    var seedData = GetCurrentSeedData();
    if (seedData != null)
        ExecutePlantSeed(seedData, request.layerIndex, request.cellPos);
    FarmToolPreview.Instance?.RemoveExecutingPreview(request.cellPos);
    _isExecutingFarming = false;
    _queuedPositions.Remove((request.layerIndex, request.cellPos));
    ProcessNextAction(); // 立即取下一个
    break;
```

整个链路在**同一帧**内同步完成：ExecutePlantSeed → ConsumeSeed → Instantiate → Initialize → SetCropData → RemoveExecutingPreview → ProcessNextAction → 下一个 ExecutePlantSeed...

**不存在"帧级竞态"**。Unity 的主线程是单线程的，同一帧内的同步调用不会产生竞态条件。

**关键发现 2：真正的幽灵占位根因 — 种子数量检查时机错误**

推演快速点击 8 次种植（袋子有 8 颗种子）的帧级时序：

```
帧 N：用户快速点击 8 次（可能跨 2-3 帧）
  → 每次点击调用 HandleUseCurrentTool → TryEnqueueSeed
  → TryEnqueueSeed 检查 farmPreview.IsValid()（预览层面有效）
  → EnqueueAction 检查 _queuedPositions 防重复（不同格子不重复）
  → 8 个请求全部入队成功

帧 N+1：ProcessNextAction 开始串行执行
  → 第1个：HasSeedRemaining() = true（slot.amount > 0）→ ExecutePlantSeed → ConsumeSeed → remaining=7
  → 第2个：HasSeedRemaining() = true → ExecutePlantSeed → ConsumeSeed → remaining=6
  → ...
  → 第7个：HasSeedRemaining() = true → ExecutePlantSeed → ConsumeSeed → remaining=1
  → 第8个：HasSeedRemaining() = true → ExecutePlantSeed → ConsumeSeed → remaining=0 → ClearSlot
```

**等等，这里有个关键问题**：

`HasSeedRemaining` 检查的是 `slot.amount > 0`，但 `ConsumeSeed` 减少的是 `bag_remaining`（种子袋内部计数），而 `ClearSlot` 只在 `remaining <= 0` 时触发。

**问题链路**：
1. 种子袋有 8 颗种子，`slot.amount = 1`（一个种子袋物品）
2. `HasSeedRemaining` 检查 `slot.amount > 0` → 始终为 true（直到 ClearSlot）
3. 但 `ConsumeSeed` 是在 `ExecutePlantSeed` 内部调用的
4. 如果 8 个请求在同一帧入队，`ProcessNextAction` 串行执行时，每次都会先检查 `HasSeedRemaining`

**实际上串行执行是安全的**——因为第 8 个执行完后 `ClearSlot` 清空了槽位，第 9 个（如果有）的 `HasSeedRemaining` 会返回 false。

**但是**，如果用户在种子袋只剩 2 颗时快速点击 5 个不同格子：
1. 入队时：5 个全部通过 `farmPreview.IsValid()` 检查（预览层面有效）
2. 执行时：第1个成功（remaining=1），第2个成功（remaining=0，ClearSlot）
3. 第3个：`HasSeedRemaining()` → `slot.IsEmpty` = true → return false → 跳过
4. 第4、5个：同样跳过

**跳过时的清理逻辑**：
```csharp
if (!HasSeedRemaining())
{
    _queuedPositions.Remove((request.layerIndex, request.cellPos));
    ProcessNextAction(); // 跳过，继续下一个
    return;
}
```

**🔴 关键 bug：跳过时只移除了 `_queuedPositions`，但没有移除队列预览！**

`AddQueuePreview` 在 `EnqueueAction` 中被调用，种子的队列预览是 SpriteRenderer 对象。当 `HasSeedRemaining` 返回 false 跳过执行时：
- `_queuedPositions.Remove` ✅ 执行了
- `RemoveExecutingPreview` ❌ 没有调用（因为 PromoteToExecutingPreview 在 ExecuteFarmAction 中，而跳过时没进入 ExecuteFarmAction）
- 队列预览的 SpriteRenderer ❌ 没有被回收

**但是**，种子的队列预览在 `AddQueuePreview` 中是通过 `activeSeedQueuePreviews` 追踪的，而 `ClearAllQueuePreviews` 会清理它们。问题是：`ProcessNextAction` 跳过后继续处理下一个，最终队列清空时 `_isProcessingQueue = false`，但 `ClearAllQueuePreviews` 只在 `ClearActionQueue` 中被调用。

**让我重新检查**：ProcessNextAction 队列为空时的清理逻辑：
```csharp
if (_farmActionQueue.Count == 0)
{
    _isProcessingQueue = false;
    _isExecutingFarming = false;
    _queuedPositions.Clear();
    _farmNavState = FarmNavState.Preview;
    return;
}
```

**🔴🔴 确认：队列清空时没有调用 ClearAllQueuePreviews！**

这意味着被跳过的种子请求的队列预览 SpriteRenderer 会残留在场景中，但因为 `queuePreviewPositions` 已被 `_queuedPositions.Clear()` 间接清理... 不对，`queuePreviewPositions` 是 FarmToolPreview 的字段，`_queuedPositions` 是 GameInputManager 的字段，它们是独立的。

**再看 AddQueuePreview 的种子分支**：
```csharp
if (type == FarmActionType.PlantSeed)
{
    var renderer = GetOrCreateSeedQueueRenderer();
    renderer.transform.position = GetCellCenterWorld(layerIndex, cellPos);
    renderer.color = new Color(1f, 1f, 1f, queuePreviewAlpha);
    renderer.enabled = true;
    activeSeedQueuePreviews.Add((cellPos, renderer));
}
queuePreviewPositions.Add(cellPos);
```

种子预览被添加到 `activeSeedQueuePreviews` 和 `queuePreviewPositions`。

**RemoveExecutingPreview 的种子分支**：
```csharp
int idx = executingSeedPreviews.FindIndex(x => x.cellPos == cellPos);
if (idx >= 0)
{
    var entry = executingSeedPreviews[idx];
    entry.renderer.enabled = false;
    seedQueuePool.Add(entry.renderer);
    executingSeedPreviews.RemoveAt(idx);
}
```

**PromoteToExecutingPreview** 会把 `activeSeedQueuePreviews` 中的条目移到 `executingSeedPreviews`。

**完整的幽灵占位推演**：

1. 种子袋剩 2 颗，用户快速点击 5 个不同格子
2. 5 个请求入队，5 个种子队列预览创建（activeSeedQueuePreviews 有 5 个条目）
3. ProcessNextAction 出队第1个 → ExecuteFarmAction → PromoteToExecutingPreview（移到 executingSeedPreviews）→ ExecutePlantSeed 成功 → RemoveExecutingPreview（回收 renderer）
4. ProcessNextAction 出队第2个 → 同上，成功，remaining=0，ClearSlot
5. ProcessNextAction 出队第3个 → HasSeedRemaining() = false → `_queuedPositions.Remove` → ProcessNextAction（跳过）
6. 第3个的队列预览：在 `activeSeedQueuePreviews` 中（没有被 Promote），renderer 仍然 enabled=true
7. 第4、5个同理
8. 队列清空 → `_queuedPositions.Clear()` → 但 `activeSeedQueuePreviews` 中仍有 3 个残留的 renderer！

**🔴🔴🔴 这就是幽灵占位的真正根因！**

不是 Instantiate 竞态条件，而是**种子数量不足时跳过执行但未清理队列预览**。

残留的 SpriteRenderer 在视觉上可能不可见（因为 alpha 较低或被其他元素遮挡），但 `queuePreviewPositions` 中的位置仍然存在，导致：
- 后续 ghost 预览在这些位置显示红色（b 层拦截）
- 玩家无法在这些位置进行任何操作

**而且**，`queuePreviewPositions` 在 ProcessNextAction 队列清空时没有被清理（只清理了 GameInputManager 的 `_queuedPositions`），所以这些幽灵位置会一直存在直到下次 `ClearAllQueuePreviews` 被调用。


**结论：❌ 锐评的根因分析事实错误**

| 锐评声称 | 代码事实 |
|---------|---------|
| Instantiate 和数据校验的帧级竞态 | Unity 主线程单线程，ExecutePlantSeed 完全同步执行，不存在竞态 |
| 预制体实例化失败或被瞬间销毁 | Instantiate 在同步链路中执行，失败时有 Destroy + 退还种子的完整回退逻辑 |
| 数据写进去了但图像丢了 | 真正的问题是：种子不足时跳过执行，但队列预览（SpriteRenderer + queuePreviewPositions）未清理 |

### 声明 4：耕地和浇水"绝对安全" 🔴

锐评声称：
> "耕地和浇水绝对安全，因为强制绑定了几百毫秒的动画（_isExecutingFarming 绝对锁），并且不使用 Instantiate，而是 Tilemap.SetTile 原子操作。有数据就必然有图像。"

**代码事实核查**：

**耕地的执行链路**：
```
ExecuteFarmAction Till → RequestAction(Crush) → _pendingTileUpdate = request
→ Update 监听动画进度 → 50% 时 ExecuteTillSoil → FarmTileManager.CreateTile
→ 动画完成 → OnFarmActionAnimationComplete → RemoveExecutingPreview → ProcessNextAction
```

**耕地是否有幽灵风险？**

耕地的 `ExecuteTillSoil` 调用 `FarmTileManager.CreateTile`，这是 Tilemap.SetTile 操作，确实是原子的——数据写入和视觉显示是同一个操作。**这一点锐评是对的。**

但耕地有另一种"幽灵"风险：**WASD 中断时的预览残留**。

分析 HandleMovement 中的 WASD 中断：
```csharp
if (hasWASD && hasActiveQueue)
{
    bool wasExecuting = _isExecutingFarming;
    ClearActionQueue();  // 清空等待队列
    
    if (!wasExecuting)
    {
        CancelFarmingNavigation();
        ToolActionLockManager.Instance?.ForceUnlock();
    }
    // else: 动画执行中 → 只清空队列
}
```

`ClearActionQueue` 调用了 `ClearAllQueuePreviews`，这会清理所有队列预览。**所以耕地的队列预览不会残留。**

但 `ClearAllQueuePreviews` 的实现需要验证——它是否正确清理了所有类型的预览？

从 memory 续27 记录和代码分析：ClearAllQueuePreviews 遍历 tillQueueTileGroups + queuePreviewPositions + activeSeedQueuePreviews，清理是完整的。

**耕地的真正风险**：不是幽灵占位，而是**执行层 tile 在 WASD 中断后的处理**。当动画执行中 WASD 中断时，`wasExecuting = true`，不调用 CancelFarmingNavigation，动画继续播放，`_pendingTileUpdate` 保留，动画完成后正常执行 ExecuteTillSoil + RemoveExecutingPreview。**这是安全的。**

**浇水的执行链路**：
```
ExecuteFarmAction Water → RequestAction(Watering) → _pendingTileUpdate = request
→ Update 监听动画进度 → 50% 时 ExecuteWaterTile → FarmTileManager.SetWatered
→ 动画完成 → OnFarmActionAnimationComplete → RemoveExecutingPreview → ProcessNextAction
```

浇水和耕地的执行链路完全一致（都是动画驱动 + _pendingTileUpdate），同样使用 Tilemap.SetTile 原子操作。**浇水也不存在幽灵占位风险。**

**但是**，用户提到的"快速点击导致左键耕地触发导航移动失效"需要单独分析：

**快速点击耕地时的帧级时序**：
1. 帧 N：用户点击左键 → HandleUseCurrentTool → TryEnqueueFarmTool → EnqueueAction → ProcessNextAction → 距离判断
2. 如果距离 > farmToolReach → StartFarmingNavigation → _farmNavState = Locked
3. 帧 N+1：用户再次快速点击左键 → HandleUseCurrentTool 检查 `_isExecutingFarming || isPerformingAction` → 如果导航中（非执行中），走到 `_farmNavState == Navigating || Locked` 分支 → TryEnqueueFromCurrentInput → 入队成功
4. 这是正常的队列入队行为，不会导致"左键耕地触发导航移动失效"

**用户描述的"左键耕地触发导航进行移动失效"更可能的场景**：
- 超快速连续点击同一格子 → 第一次入队成功，后续被 `_queuedPositions` 防重复拦截 → 但 HandleUseCurrentTool 的 return 阻止了后续逻辑 → 用户感觉"点击无反应"
- 或者：种子幽灵占位后，`queuePreviewPositions` 残留导致 b 层拦截 → ghost 预览显示红色 → `farmPreview.IsValid()` 返回 false → TryEnqueueSeed/TryEnqueueFarmTool 直接 return → 用户无法在该位置进行任何操作

**结论：⚠️ 部分正确**

| 锐评声称 | 代码事实 |
|---------|---------|
| 耕地和浇水绝对安全 | ✅ 耕地和浇水确实不存在"数据有但视觉无"的幽灵占位 |
| 动画锁保护 | ✅ `_isExecutingFarming` + ToolActionLockManager 双重保护有效 |
| Tilemap.SetTile 原子操作 | ✅ 正确，数据和视觉是同一操作 |
| "百分之一万放心" | ⚠️ 过于绝对。耕地和浇水虽无幽灵占位，但快速点击仍可能导致用户体验问题（如种子幽灵占位后的连锁影响） |

### 问题 A：种子数量不足时的边界情况

**用户原话**：
> 当前内容是一个袋子8个种子，对于快速点击的时候会出现多种的情况，但是这个多种不会显示，可是却占了坑位

**代码推演**（袋子 8 颗种子，快速点击 10 个不同格子）：

1. 入队阶段（可能跨 1-3 帧）：
   - TryEnqueueSeed 检查 `farmPreview.IsValid()` → 每个格子都有效
   - EnqueueAction 检查 `_queuedPositions` → 不同格子不重复
   - 10 个请求全部入队成功
   - AddQueuePreview 创建 10 个种子 SpriteRenderer

2. 执行阶段（同一帧内串行）：
   - 第1-8个：HasSeedRemaining() = true → ExecutePlantSeed 成功 → ConsumeSeed 递减
   - 第8个执行后：remaining=0 → ClearSlot → 槽位清空
   - 第9个：HasSeedRemaining() → slot.IsEmpty = true → return false → **跳过**
   - 第10个：同样跳过

3. 跳过时的清理：
   ```csharp
   _queuedPositions.Remove((request.layerIndex, request.cellPos));
   ProcessNextAction(); // 跳过，继续下一个
   ```
   - ✅ `_queuedPositions` 移除了
   - ❌ `queuePreviewPositions` 没有移除（这是 FarmToolPreview 的字段）
   - ❌ `activeSeedQueuePreviews` 中的 SpriteRenderer 没有回收
   - ❌ 没有调用 RemoveExecutingPreview（因为没进入 ExecuteFarmAction，没有 Promote）

4. 队列清空时：
   ```csharp
   _isProcessingQueue = false;
   _isExecutingFarming = false;
   _queuedPositions.Clear(); // GameInputManager 的字段
   _farmNavState = FarmNavState.Preview;
   ```
   - ❌ 没有调用 ClearAllQueuePreviews
   - ❌ FarmToolPreview 的 queuePreviewPositions 仍包含第9、10个格子
   - ❌ activeSeedQueuePreviews 仍有 2 个残留 renderer

**后果**：
- 第9、10个格子的 `queuePreviewPositions` 残留 → b 层拦截生效 → ghost 预览显示红色
- 残留的 SpriteRenderer 可能可见也可能不可见（取决于 alpha 和遮挡）
- 用户无法在这些格子进行任何农田操作（耕地、浇水、播种都被 b 层拦截）
- **这就是用户描述的"幽灵占位"现象**

**🔴 这是一个确认的 bug，根因是 ProcessNextAction 跳过种子时未清理 FarmToolPreview 的队列预览数据。**

### 问题 B：快速点击导致左键耕地触发导航移动失效

**用户原话**：
> 种子引发的怪异 bug 还有玩家无法进行左键耕地触发导航进行移动

**分析**：这是问题 A 的连锁反应。

当种子幽灵占位发生后：
1. `queuePreviewPositions` 残留了幽灵格子
2. 用户切换到锄头 → UpdateHoePreview 执行
3. 鼠标悬停在幽灵格子上 → `queuePreviewPositions.Contains(cellPos)` = true

7. **用户感觉"左键无反应"，无法耕地也无法触发导航**

**更严重的是**：如果用户在多个格子产生了幽灵占位，这些格子会形成一片"死区"，所有农田操作都被拦截。

**🔴 确认：这不是独立 bug，而是种子幽灵占位的连锁影响。修复幽灵占位即可解决。**

### 问题 C：浇水是否有幽灵情况

**代码推演**（快速点击浇水 10 个不同格子）：

1. 入队阶段：TryEnqueueFarmTool → EnqueueAction → AddQueuePreview（Water 分支，Tilemap.SetTile）
2. 执行阶段：每个请求都走动画流程（Watering 动画 → ExecuteWaterTile → OnFarmActionAnimationComplete）
3. 浇水没有"数量不足"的概念（水壶无限使用），不存在跳过执行的场景
4. 每个请求都会完整走完：ExecuteFarmAction → 动画 → ExecuteWaterTile → RemoveExecutingPreview

**浇水的二次验证**：ProcessNextAction 中浇水没有二次验证分支（只有 PlantSeed 和 Harvest 有）。所以浇水请求不会被跳过。

**但是**，浇水有一个边界情况：`tileData.wateredToday` 检查。如果同一格子被重复入队（理论上被 `_queuedPositions` 防重复拦截），或者在队列执行期间该格子已被浇水（比如其他系统触发），`SetWatered` 可能返回 false。但这不会导致幽灵占位，因为 OnFarmActionAnimationComplete 无条件执行 RemoveExecutingPreview。

**结论：✅ 浇水不存在幽灵占位风险。**

### 问题 D：耕地是否有幽灵情况

**代码推演**（快速点击耕地 10 个不同格子）：

1. 入队阶段：TryEnqueueFarmTool → EnqueueAction → AddQueuePreview（Till 分支，完整 1+8 tile）
2. 执行阶段：每个请求走动画流程（Crush 动画 → ExecuteTillSoil → OnFarmActionAnimationComplete）
3. 耕地没有"数量不足"的概念（锄头无限使用），不存在跳过执行的场景
4. ProcessNextAction 中耕地没有二次验证分支

**耕地的边界情况**：`CanTillAt` 在执行时可能返回 false（比如该格子在队列执行期间被其他系统耕作了）。但 ExecuteTillSoil 返回 false 不影响后续流程——OnFarmActionAnimationComplete 无条件执行 RemoveExecutingPreview。

**结论：✅ 耕地不存在幽灵占位风险。**

### 问题 E：耕地和浇水是否会串流互相影响

**代码推演**（交替快速点击耕地和浇水）：

1. 用户持有锄头 → 点击格子 A → TryEnqueueFarmTool → type=Till → 入队
2. 用户切换到水壶 → 点击格子 B → TryEnqueueFarmTool → type=Water → 入队
3. 队列中：[Till@A, Water@B]
4. ProcessNextAction 串行执行：先 Till@A（Crush 动画），完成后 Water@B（Watering 动画）

**串流风险分析**：
- `_isExecutingFarming` 在 ExecuteFarmAction 开头设为 true，在 OnFarmActionAnimationComplete 末尾设为 false → 串行保护有效
- `_pendingTileUpdate` 在 ExecuteFarmAction 中设置，在 OnFarmActionAnimationComplete 中清空 → 不会串流
- 预览系统：PromoteToExecutingPreview 在 ExecuteFarmAction 中调用，RemoveExecutingPreview 在完成后调用 → 每个请求独立管理

**唯一的串流风险**：如果用户在耕地动画执行中切换了工具（锄头→水壶），UpdatePreviews 会切换到 UpdateWateringPreview，但这不影响队列中已有的请求——队列请求的 type 在入队时已确定。

**结论：✅ 耕地和浇水不会串流互相影响。队列系统的串行执行保证了操作隔离。**

## 五、真正的 bug 清单

### Bug 1（🔴 严重）：种子数量不足时队列预览残留

**根因**：ProcessNextAction 中 `HasSeedRemaining()` 返回 false 跳过执行时，只移除了 `_queuedPositions`（GameInputManager），未清理 FarmToolPreview 的 `queuePreviewPositions` 和 `activeSeedQueuePreviews`。

**影响**：
- 残留的 `queuePreviewPositions` 导致 b 层拦截，所有农田操作在幽灵格子上失效
- 残留的 SpriteRenderer 可能可见（低 alpha 半透明作物图标）
- 连锁影响：切换到锄头/水壶后，幽灵格子显示红色，无法操作

**修复方向**：ProcessNextAction 跳过种子时，需要同时清理 FarmToolPreview 的队列预览：
```
方案 A：跳过时调用 FarmToolPreview.RemoveQueuePreview(cellPos)（如果有此方法）
方案 B：跳过时调用 FarmToolPreview.RemoveExecutingPreview(cellPos)（但种子没有被 Promote）
方案 C：在 ProcessNextAction 队列清空时调用 ClearAllQueuePreviews
```

最稳妥的方案是 **方案 C**：在 ProcessNextAction 队列清空分支中增加 `FarmToolPreview.Instance?.ClearAllQueuePreviews()`。这样无论什么原因导致的残留都能被清理。

但方案 C 有一个问题：如果队列正常清空（所有请求都成功执行），ClearAllQueuePreviews 会清理已经被 RemoveExecutingPreview 处理过的数据——不过这是安全的（重复清理不会出错）。

**更精确的方案**：在 ProcessNextAction 跳过种子的分支中，直接清理该格子的队列预览：

```csharp
case FarmActionType.PlantSeed:
    if (!HasSeedRemaining())
    {
        _queuedPositions.Remove((request.layerIndex, request.cellPos));
        // 🔴 修复：清理 FarmToolPreview 的队列预览
        FarmToolPreview.Instance?.RemoveSeedQueuePreview(request.cellPos);
        ProcessNextAction();
        return;
    }
    break;
```

需要在 FarmToolPreview 中新增 `RemoveSeedQueuePreview(Vector3Int cellPos)` 方法，从 `activeSeedQueuePreviews` 中找到并回收 renderer，同时从 `queuePreviewPositions` 中移除。

**同时**，ProcessNextAction 队列清空时也应该调用 ClearAllQueuePreviews 作为兜底安全网。

### Bug 2（🟡 中等）：ProcessNextAction 队列清空时未清理 FarmToolPreview 状态

**根因**：ProcessNextAction 队列清空分支只清理了 GameInputManager 的状态，没有通知 FarmToolPreview。

**当前代码**：
```csharp
if (_farmActionQueue.Count == 0)
{
    _isProcessingQueue = false;
    _isExecutingFarming = false;
    _queuedPositions.Clear();
    _farmNavState = FarmNavState.Preview;
    return;
}
```

**缺少**：`FarmToolPreview.Instance?.ClearAllQueuePreviews()` 或至少 `queuePreviewPositions.Clear()`

**影响**：即使没有种子幽灵占位，如果未来有其他类型的请求被跳过（比如 Harvest 的二次验证失败），同样会导致队列预览残留。

**修复方向**：在队列清空分支增加 ClearAllQueuePreviews 调用。
