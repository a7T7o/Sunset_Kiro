## 二、事实核查

### 声明 1："Bug 1/Bug 2 纯粹只是修 UI 视觉残留"

**锐评原文**：
> Kiro 的这两个修复，纯粹只是在修"UI 视觉残留"！它完全没有理解你说的"幽灵占位"的核心痛点！幽灵占位不仅是预览没消失，更恶劣的是"坑位被占了（数据被写死），但是东西没种出来（实例化失败）"。

**代码事实核查**：

Bug 1 修复的是 `queuePreviewPositions` 残留。这个字段不仅仅是"UI 视觉"——它是 b 层拦截的数据源：

```
UpdateHoePreview: if (queuePreviewPositions.Contains(cellPos)) → canTill = false
UpdateWateringPreview: if (queuePreviewPositions.Contains(cellPos)) → canWater = false  
UpdateSeedPreview: if (queuePreviewPositions.Contains(cellPos)) → canPlant = false
```

`queuePreviewPositions` 残留导致的后果：
- 所有农田操作在幽灵格子上被 b 层拦截 → `IsValid() = false`
- 用户无法在这些格子进行任何操作（耕地、浇水、播种）
- 这不是"UI 视觉残留"，而是**数据层拦截残留**

至于"坑位被占了但东西没种出来"——这个描述本身就不准确。代码推演：

- 种子袋剩 2 颗，用户快速点击 5 个不同格子
- 5 个请求入队，5 个种子队列预览创建
- 执行第1个：成功种植，remaining=1
- 执行第2个：成功种植，remaining=0，ClearSlot
- 第3-5个：HasSeedRemaining() = false → 跳过
- **跳过时**：`_queuedPositions.Remove` ✅ 但 `queuePreviewPositions` ❌ 未清理

"坑位被占了"指的是 `queuePreviewPositions` 残留（b 层拦截数据），不是 FarmTileData 的数据层。FarmTileData 中这些格子根本没有被写入任何数据——因为 ExecutePlantSeed 根本没有被调用。

**结论：❌ 事实错误**

Bug 1 修复的不是"UI 视觉残留"，而是 b 层拦截数据（`queuePreviewPositions`）的残留。这个残留直接导致所有农田操作在幽灵格子上失效。修复 Bug 1 就是修复"坑位被占"的根因。

### 声明 2："PlantSeed 是零耗时操作，导致帧级竞态冲突"

**锐评原文**：
> 种子的播种（PlantSeed）是零耗时的！只要它还留在 _farmActionQueue 这个原本为动画锁设计的队列里，玩家连点产生的高频零耗时出队，就会疯狂调用 ExecutePlantSeed，导致 CanPlant 状态写入与 Instantiate 实例化发生严重的竞态冲突。这就是为什么坑位被占了，但预制体没生成！

**代码事实核查**：

**PlantSeed 确实是"零耗时"的**——这一点 ✅ 正确。ExecuteFarmAction 的 PlantSeed 分支没有动画，直接同步执行：

```csharp
case FarmActionType.PlantSeed:
    var seedData = GetCurrentSeedData();
    if (seedData != null)
        ExecutePlantSeed(seedData, request.layerIndex, request.cellPos);
    FarmToolPreview.Instance?.RemoveExecutingPreview(request.cellPos);
    _isExecutingFarming = false;
    _queuedPositions.Remove((request.layerIndex, request.cellPos));
    ProcessNextAction(); // 立即取下一个
    break;
```

**但"帧级竞态冲突"不存在**——这一点 ❌ 事实错误。

Unity 主线程是单线程的。`ProcessNextAction → ExecuteFarmAction → ExecutePlantSeed → ProcessNextAction → ExecuteFarmAction → ExecutePlantSeed...` 这个递归调用链在**同一帧内同步串行执行**。

每次 ExecutePlantSeed 执行时：
1. `tileData.CanPlant()` 检查 → 如果已有作物返回 false → 不种植
2. `ConsumeSeed` 消耗种子 → 原子操作
3. `Instantiate` 创建 GameObject → 同步操作
4. `tileData.SetCropData` 写入数据 → 同步操作

第 N 次 ExecutePlantSeed 完成后，数据层已经更新完毕，第 N+1 次的 `CanPlant()` 检查能看到最新状态。**不存在"CanPlant 状态写入与 Instantiate 实例化的竞态冲突"**。

**"坑位被占了但预制体没生成"的真正原因**：

不是竞态冲突，而是**种子数量不足时跳过执行但未清理队列预览**（Bug 1）。被跳过的格子：
- FarmTileData 中没有任何数据写入（ExecutePlantSeed 根本没被调用）
- 但 `queuePreviewPositions` 残留 → b 层拦截 → 用户感觉"坑位被占了"
- 没有 Instantiate 调用 → 没有 GameObject → 用户感觉"东西没种出来"

**结论：❌ 核心论点事实错误**

| 锐评声称 | 代码事实 |
|---------|---------|
| 高频零耗时出队导致竞态 | Unity 单线程，同步串行执行，无竞态 |
| CanPlant 与 Instantiate 冲突 | 每次 ExecutePlantSeed 完成后数据层已更新，下一次能看到最新状态 |
| 坑位被占但预制体没生成 | 根因是 Bug 1（队列预览未清理），不是竞态 |
