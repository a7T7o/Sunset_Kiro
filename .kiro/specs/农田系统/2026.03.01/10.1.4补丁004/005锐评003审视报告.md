## 二、事实核查

### 声明 1："种子没有动画，放在为动画锁设计的队列里是天然的代码毒瘤"

**代码事实**：

`ExecuteFarmAction` 中 PlantSeed 分支确实与其他类型完全不同：

```
Till/Water/RemoveCrop → 播放动画 → 等待 OnActionComplete 回调
Harvest → 播放收集动画 → 等待 OnCollectAnimationComplete 回调
PlantSeed → 无动画 → 直接执行 → 立即 ProcessNextAction()
```

PlantSeed 是唯一一个在 `ExecuteFarmAction` 中同步完成并立即递归调用 `ProcessNextAction()` 的类型。FIFO 队列的核心价值是串行化有动画的操作（防止动画冲突），种子不需要这个机制。

**结论：✅ 事实正确**

种子走 FIFO 队列确实是架构错位——它不需要动画锁、不需要串行等待、不需要执行预览保护。

### 声明 2："长痛不如短痛，必须立刻剥离"

**客观评估**：方向正确，但"立刻"需要权衡。见第三章全局分析。

**结论：⚠️ 方向正确，时机需权衡**

### 种子走 FIFO 的代价（当前问题）

1. **Bug 1（🔴严重）**：`ProcessNextAction` 跳过种子时 `queuePreviewPositions` 未清理 → b 层拦截残留 → 幽灵占位
2. **Bug 2（🟡中等）**：队列清空时未调用 `ClearAllQueuePreviews` → 种子预览残留
3. **架构复杂度**：FarmToolPreview 需要维护两套完全不同的预览系统（Tilemap 系 vs SpriteRenderer 系），`AddQueuePreview`/`RemoveExecutingPreview`/`ClearAllQueuePreviews` 都需要分支处理
4. **执行路径特殊**：PlantSeed 是唯一同步递归调用 `ProcessNextAction()` 的类型，与其他类型的异步回调模式不一致

### 种子走放置系统的可行性

**已有基础设施**：
- `PlacementManager` 已有完整的放置流程：Preview → Locked → Navigating → Executing
- `PlacementPreview` 已有预览显示、颜色控制、锁定/解锁机制
- `PlacementValidator` 已有障碍物检测
- `HotbarSelectionService` 已有 `isPlaceable` → `EnterPlacementMode` 路由
- 树苗（TreeControllerV2）已经是放置系统的成功案例

**需要适配的部分**：
- `PlacementManager.EnterPlacementMode` 中移除 SeedData 拦截
- 种子的放置验证需要额外检查：耕地存在（`tileData != null`）、可种植（`CanPlant()`）、季节匹配
- 种子放置后的执行逻辑（消耗种子、创建作物 GameObject、设置 CropData）需要接入 PlacementManager 的执行流程
- 种子预览需要显示作物第一阶段 sprite（当前 FarmToolPreview 已实现，需迁移到 PlacementPreview 或新建种子专用预览）

**复杂度评估**：中等。核心逻辑（`ExecutePlantSeed`）可以直接复用，主要工作是路由改造和预览迁移。

### 保持现状（先修 Bug）的利弊

**优势**：
- 改动极小：ProcessNextAction 中增加种子跳过时的清理代码（约 5 行）
- 立即解决幽灵占位问题
- 不引入新风险

**劣势**：
- 在架构错位的代码上打补丁
- FarmToolPreview 继续维护两套预览系统
- 后续每次修改队列逻辑都要考虑种子的特殊分支
- 种子的 b 层拦截逻辑与耕地/浇水混在一起，增加理解成本

### 立即剥离（模块R）的利弊

**优势**：
- 一步到位，彻底消除种子在 FIFO 中的所有特殊分支
- FarmToolPreview 简化（移除所有种子相关字段和方法）
- GameInputManager 简化（移除 TryEnqueueSeed、PlantSeed 枚举值的所有分支）
- 放置系统未来壮大时，种子自然受益（用户原话："后续放置系统肯定会更加壮大"）
- 与树苗放置逻辑统一，降低认知负担

**劣势**：
- 工作量中等（多文件改动）
- 需要设计种子在放置系统中的验证逻辑（耕地检查、季节检查）
- 如果引入新 bug，调试范围更大


### 全局视角：放置系统的未来

用户强调"后续放置系统肯定会更加壮大因为会处理更多情况和内容"。从全局看：

- 放置系统当前处理：树苗、箱子（待开发：工作台、装饰物）
- 种子接入后，放置系统处理的物品类型更丰富，验证逻辑更通用
- 种子的"需要耕地"验证可以作为放置系统的一种验证规则（类似树苗的"需要空地"验证）
- 未来如果有其他需要特定条件的放置物品（如花盆需要桌面），种子的验证模式可以复用

### 我的最终结论

**种子走 FIFO 队列本质上是错的。**

理由：
1. FIFO 队列的核心价值是串行化有动画的操作。种子没有动画，不需要串行。
2. 种子在队列中是唯一的"同步递归"类型，与其他类型的异步回调模式不一致，是 bug 的温床（Bug 1/Bug 2 就是证据）。
3. 种子的预览系统（SpriteRenderer）与耕地/浇水的预览系统（Tilemap）完全不同，强行塞在同一个 FarmToolPreview 中增加了不必要的复杂度。
4. 放置系统已有完整的基础设施，种子接入的适配工作量可控。

**建议：执行模块R（种子剥离），但分两步走。**

| 步骤 | 内容 | 理由 |
|------|------|------|
| 步骤 1 | 先修 Bug 1/Bug 2（5 行代码） | 立即解决幽灵占位，给模块R创造干净的起点 |
| 步骤 2 | 执行模块R（种子剥离到放置系统） | 彻底消除架构错位 |

为什么建议先修 Bug 再剥离，而不是直接剥离？
- Bug 1/Bug 2 的修复代码极小（ProcessNextAction 中增加清理），即使种子后续被剥离，这几行代码也不会造成任何负担
- 修复后可以立即验收幽灵占位问题是否解决，确认根因分析正确
- 模块R 是较大的重构，如果重构过程中引入新问题，有 Bug 修复作为对照基线更容易定位
- 但如果用户觉得"先修再剥离"是多此一举，直接剥离也完全可行——因为剥离本身就会移除 Bug 1/Bug 2 的触发路径

**无论选哪种方式，种子最终都应该离开 FIFO 队列。这不是"要不要做"的问题，而是"什么时候做"的问题。用户说得对——放置系统未来会壮大，种子应该在那里。**
