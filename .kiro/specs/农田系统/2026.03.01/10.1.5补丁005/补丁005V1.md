## 二、任务A：修复房子不标红视觉路由 bug

### 2.1 问题根因（代码事实）

文件：`FarmToolPreview.cs` → `UpdateHoePreview` 方法

当前逻辑（第435-475行）：

```csharp
bool hasObstacle = PlacementValidator.HasFarmingObstacle(cellCenter);  // 房子 → true
bool canTill = FarmTileManager.Instance.CanTillAt(layerIndex, cellPos); // 有地面tile → true
bool isValid = !hasObstacle && (canTill || hasCrop);                    // false && true = false
```

视觉路由三分支（第490-640行）：

```
分支1: if (canTill && FarmlandBorderManager.Instance != null)  → 1+8 预览
分支2: else if (hasCrop)                                       → 空白（有作物）
分支3: else                                                    → 红方框
```

问题：房子下有地面 tile → `canTill=true` → 进入分支1（1+8 预览），跳过分支3（红方框）。
同时 Shader 染色也基于 `canTill`（第643行），显示绿色。

结果：指着房子时显示绿色 1+8 耕地预览，但实际 `isValid=false` 不可耕种。

### 2.2 修复方案

将视觉路由的第一个分支条件从 `canTill` 改为 `isValid && !hasCrop`：

```
修改前: if (canTill && FarmlandBorderManager.Instance != null)
修改后: if (isValid && !hasCrop && FarmlandBorderManager.Instance != null)
```

Shader 染色同步修改：

```
修改前: if (canTill)
修改后: if (isValid && !hasCrop)
```

这样当 `hasObstacle=true` 时，`isValid=false`，不会进入 1+8 预览分支，而是落入分支3 显示红方框。

### 2.3 影响分析

修改范围：仅 `FarmToolPreview.cs` 的 `UpdateHoePreview` 方法内部。

| 场景 | 修改前 | 修改后 | 正确性 |
|------|--------|--------|--------|
| 空地（可耕种） | canTill=T, hasObstacle=F → 1+8绿色 | isValid=T → 1+8绿色 | ✅ 不变 |
| 房子下方 | canTill=T, hasObstacle=T → 1+8绿色 ❌ | isValid=F → 红方框 | ✅ 修复 |
| 已耕种（有作物） | hasCrop=T → 空白 | hasCrop=T → 空白 | ✅ 不变 |
| 石头/树木 | canTill=F → 红方框 | isValid=F → 红方框 | ✅ 不变 |
| 已在队列中 | canTill=F(b层拦截) → 红方框 | isValid=F → 红方框 | ✅ 不变 |

关键验证：b 层拦截（`queuePreviewPositions.Contains`）在 `isValid` 计算之前将 `canTill` 设为 false，所以 `isValid` 也会是 false。逻辑链完整。

### 2.4 涉及文件

| 文件 | 修改点 |
|------|--------|
| `FarmToolPreview.cs` | `UpdateHoePreview` 视觉路由分支条件 + Shader 染色条件 |

## 三、任务B：种子从 FIFO 彻底剥离并接入放置系统

### 3.1 架构决策依据

种子走 FIFO 队列是架构错位，代码事实如下：

1. FIFO 队列的核心价值 = 串行化有动画的操作（耕地/浇水/收获都有动画 → 异步回调 `OnFarmActionAnimationComplete`）
2. 种子是 FIFO 中唯一的同步递归类型：`ExecuteFarmAction` PlantSeed 分支无动画 → 直接执行 → `ProcessNextAction()` 递归
3. 种子的预览系统（SpriteRenderer 对象池）与耕地/浇水的预览系统（Tilemap）完全不同
4. 种子在 FIFO 中有 14 个触点，每个都需要特殊分支处理

放置系统已有完整基础设施：状态机（Preview → Locked → Navigating → Executing）、导航、预览、验证、扣除物品。树苗是成功案例。

### 3.3 种子接入放置系统的适配方案

#### 3.3.1 数据层：SeedData 标记为可放置

文件：`SeedData.cs`

当前状态：`SeedData` 继承自 `ItemData`，`isPlaceable` 默认为 false。

修改方案：在 `SeedData` 的构造或 `OnValidate` 中强制设置 `isPlaceable = true`，或者在 SO 资产上手动勾选。

考虑点：`isPlaceable = true` 会让 `HotbarSelectionService` 自动调用 `PlacementManager.EnterPlacementMode`，这正是我们想要的路由。

#### 3.3.2 路由层：移除 SeedData 拦截

文件：`PlacementManager.cs` → `EnterPlacementMode`

当前代码（第303-309行）：
```csharp
if (item is SeedData)
{
    // SeedData 不通过放置系统处理
    return;
}
```

修改：删除这个 if 块。

#### 3.3.3 验证层：种子专用放置验证

种子的放置验证与树苗不同，需要检查：
1. 目标位置有耕地数据（`tileData != null`）
2. 耕地可种植（`tileData.CanPlant()`）
3. 无障碍物（`!HasFarmingObstacle`）
4. 季节匹配（`IsCorrectSeason`）

实现方案有两种选择：

方案α：在 `PlacementValidator` 中新增种子验证方法
- 新增 `ValidateSeedPlacement(SeedData, Vector3, int layerIndex)` 方法
- `PlacementManager.UpdatePreview` 和 `ExecutePlacement` 中检测 `SeedData` 类型走专用验证

方案β：利用 `PlaceableItemData.CanPlaceAt` 虚方法
- `SeedData` 重写 `CanPlaceAt(Vector3 position)` 方法
- 在内部调用 `FarmTileManager` 检查耕地状态
- 放置系统无需修改验证逻辑，自动走 `CanPlaceAt`

推荐方案α，原因：
- 种子验证需要 `layerIndex`（楼层索引），`CanPlaceAt` 只接收 `Vector3`，信息不足
- 种子验证逻辑与农田系统强耦合（`FarmTileManager`、`FarmTileData`），放在 `PlacementValidator` 更合适
- 保持 `SeedData` 作为纯数据类的职责

#### 3.3.4 预览层：种子预览迁移

当前种子预览由 `FarmToolPreview.UpdateSeedPreview` 实现，使用两个 SpriteRenderer：
- `seedGridRenderer`：底部格子方框（绿/红）
- `seedPreviewRenderer`：作物第一阶段 sprite + 底部对齐偏移

放置系统的预览由 `PlacementPreview` 实现，使用 `PlacementGridCell`（格子方框）+ `itemPreviewRenderer`（物品预览）。

迁移方案：
- `PlacementPreview.Show(ItemData item)` 检测到 `SeedData` 时，使用作物第一阶段 sprite 作为预览图
- 底部对齐偏移逻辑复用 `AlignSpriteBottom` 模式
- 格子方框颜色由 `PlacementGridCell` 的 valid/invalid 状态控制（已有机制）

关键适配点：
- `PlacementPreview` 当前通过 `item.placementPrefab` 获取预览 sprite
- 种子没有 `placementPrefab`，需要从 `seedData.cropPrefab` 的 `CropController.GetFirstStageSprite()` 获取
- 需要在 `PlacementPreview.Show` 中增加 SeedData 特殊处理分支

#### 3.3.5 执行层：种子专用放置执行

当前 `PlacementManager.ExecutePlacement` 的流程：
1. 验证快照 → 2. `CanPlaceAt` 检查 → 3. 保存数据 → 4. `InstantiatePlacementPrefab` → 5. 同步 Layer → 6. 设置 SortingOrder → 7. 扣除物品 → 8. 添加历史 → 9. 播放音效/特效

种子的执行逻辑（当前 `ExecutePlantSeed`）：
1. 获取耕地数据 → 2. `CanPlant` 检查 → 3. 季节检查 → 4. `SeedBagHelper.ConsumeSeed`（消耗种子） → 5. `Instantiate(cropPrefab)` → 6. `CropController.Initialize` → 7. `tileData.SetCropData`

两者差异：
- 种子不走 `InstantiatePlacementPrefab`（不使用 `placementPrefab`），而是 `Instantiate(seedData.cropPrefab)`
- 种子的物品扣除走 `SeedBagHelper`（种子袋保质期链路），不是简单的 `inventory.RemoveItem`
- 种子放置后需要 `CropController.Initialize` + `tileData.SetCropData`
- 种子不需要 `SyncLayerToPlacedObject`（作物的 Layer 由 `CropController.Initialize` 内部处理）
- 种子不需要放置历史（不支持撤销）

实现方案：在 `PlacementManager.ExecutePlacement` 中增加 `SeedData` 专用分支：

```
if (currentPlacementItem is SeedData seedData)
{
    ExecuteSeedPlacement(seedData);  // 种子专用执行路径
    return;
}
// ... 原有通用放置逻辑不变
```

`ExecuteSeedPlacement` 方法的核心逻辑直接从 `GameInputManager.ExecutePlantSeed` 迁移，包括：
- `FarmTileManager` 获取耕地数据
- `SeedBagHelper.ConsumeSeed` 消耗种子
- `Instantiate(cropPrefab)` 实例化作物
- `CropController.Initialize` 初始化
- `tileData.SetCropData` 更新耕地数据

#### 3.3.6 导航层：复用放置系统导航

当前种子的导航由 `GameInputManager.TryPlantSeed` 中的远距离分支处理（`StartFarmingNavigation`）。

放置系统已有完整导航流程：
- `OnLeftClick` → `LockPreviewPosition` → `StartNavigation` → `OnNavigationReached` → `ExecutePlacement`

种子接入后自动复用此流程，无需额外适配。

#### 3.3.7 连续放置：种子用完检测

当前 FIFO 中的种子用完检测：`HasSeedRemaining`（ProcessNextAction 二次验证）。

放置系统中的物品用完检测：`HasMoreItems`（ExecutePlacement 末尾）。

`HasMoreItems` 当前实现：
```csharp
private bool HasMoreItems()
{
    // 使用快照中的槽位索引检查
    var slot = inventoryService.GetSlot(currentSnapshot.slotIndex);
    return !slot.IsEmpty && slot.itemId == currentSnapshot.itemId;
}
```

种子的特殊性：种子袋的 `amount` 可能为 1 但内部 `remaining` 为 0（种子袋用完但物品还在槽位）。需要在 `HasMoreItems` 中增加种子袋检测：

```
if (currentPlacementItem is SeedData)
{
    // 检查种子袋是否还有剩余种子
    var seedItem = inventoryService.GetInventoryItem(currentSnapshot.slotIndex);
    return seedItem != null && !seedItem.IsEmpty 
        && SeedBagHelper.GetRemaining(seedItem) > 0;
}
```

### 3.4 FIFO 清理后的副作用分析

种子从 FIFO 移除后，以下 bug 自然消失（无需单独修复）：
- Bug 1：`queuePreviewPositions` 残留 → 幽灵占位（种子不再进入 queuePreviewPositions）
- Bug 2：种子预览 SpriteRenderer 残留（种子不再使用 FarmToolPreview 的 SpriteRenderer 对象池）

FIFO 队列清理后的纯净度提升：
- `FarmActionType` 枚举中 `PlantSeed` 不再被使用（可保留避免序列化问题，或标记 Obsolete）
- `ProcessNextAction` 不再需要种子二次验证分支
- `ExecuteFarmAction` 不再需要种子同步递归分支
- `FarmToolPreview` 不再维护两套预览系统（Tilemap + SpriteRenderer）
